# HealthTick 发布指南

本文档介绍了如何将 HealthTick 打包为正式版并发布。

---

## 自动化发布流程

项目根目录下提供了 `release.sh` 脚本，可一键完成编译、签名、打包 dmg、Git 打标签以及发布到 GitHub/Gitee 的全过程。

### 1. 准备工作

在执行发布脚本前，请确保满足以下条件：

- **macOS 环境**：脚本依赖 `hdiutil`、`codesign` 等系统工具。
- **Swift 工具链**：本地已安装 Swift 且 `swift --version` 可用。
- **权限与证书**：
  - 脚本默认执行 Ad-hoc 签名（`--sign -`）。
  - 若需发布到线上仓库，需配置好 GitHub CLI (`gh`) 登录状态，并设置环境变量 `GITEE_TOKEN`。

### 2. 发布步骤

#### 第一步：更新版本号

修改 `Sources/Info.plist` 中的版本信息：
- `CFBundleShortVersionString`：展示给用户的版本号（如 `1.6.12`）。
- `CFBundleVersion`：内部 Build 号，每次发布必须递增。

#### 第二步：执行脚本

在终端运行：
```bash
./release.sh
```

### 3. 脚本执行细节

脚本会按顺序执行以下六个步骤：

1.  **编译二进制 (Binaries)**：分别构建 `arm64` (Apple Silicon) 和 `x86_64` (Intel) 的 Release 架构产物。
2.  **组装 App Bundle**：在 `/tmp` 目录下创建标准的 `.app` 结构，注入图标、资源文件。
3.  **创建 DMG**：打包为两个架构的磁盘镜像文件。
4.  **Git 推送**：自动将所有未提交的修改 commit，并打上 `v[版本号]` 的 Tag 推送到远端仓库。
5.  **GitHub Release**：调用 `gh` 工具创建 Release 并上传 DMG 附件。
6.  **Gitee Release**：通过 API 在 Gitee 创建发行版并上传附件（需配置 Token）。

## 手动打包流程

如果只需在本地生成正式版 `.app` 而不执行脚本中的自动发布逻辑，可以参考以下步骤：

### 1. Release 编译

```bash
# 构建全平台（通用）二进制
swift build -c release --arch arm64 --arch x86_64
```

### 2. 获取产物

编译后的二进制文件位于：
`.build/apple/Products/Release/HealthTick`

### 3. 生成安装包

将生成的二进制文件放入准备好的 `.app/Contents/MacOS/` 目录下，并补充 `Info.plist` 和 `Resources` 资源。详情可参考 `release.sh` 中的 `[2/6] Packaging apps` 逻辑。

## 常见问题

### 提示 "Tag already exists"

如果脚本报错 `Error: tag vXXX already exists`，说明你忘记在 `Info.plist` 中增加版本号，或者该版本已经发布过。

### 云端发布失败

如果脚本在第 5、6 步失败，通常是由于网络连接 GitHub 失败或 Token 失效。此时 DMG 文件已经成功生成在 `/tmp/health-tick-release-[版本号]/` 目录下，你可以手动上传到 GitHub 仓库对应的 Release 页面。

### codesign 报错

脚本使用 Ad-hoc 签名。如果你的系统由于特殊安全策略拒绝此签名，你可能需要使用开发者 ID 证书重新签名，或者在系统设置中手动允许运行。
